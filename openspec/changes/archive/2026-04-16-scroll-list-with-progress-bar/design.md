## Context

当前专辑列表 UI（`App.vue`）使用传统分页模式：
- 后端 `album-service.ts` 的 `queryAlbums` 支持 `page` 和 `pageSize` 参数
- 前端每次请求一页数据，通过底部分页控件切换页面
- 表格使用标准 `<table>` 结构，表头和内容一起渲染

目标是改为无限滚动 + 固定表头 + 自定义进度条的现代化列表体验。

## Goals / Non-Goals

**Goals:**
- 用无限滚动替代分页，提供连续浏览体验
- 固定表头，让用户在滚动时始终能看到列名
- 提供可拖动的自定义滚动进度条
- 进度条有视觉反馈（滚动时明显，静止时淡化）

**Non-Goals:**
- 不做虚拟滚动（Virtual Scrolling）—— 专辑数量预计在千级别，完整 DOM 渲染可接受
- 不改变后端 API —— 仍使用现有分页查询接口
- 不添加"回到顶部"按钮 —— 用户可通过拖动进度条实现

## Decisions

### 1. 数据加载策略：增量加载而非一次性加载

**选择**：滚动到底部附近时，追加加载下一页数据到现有列表

**理由**：
- 初始加载快（只加载第一页）
- 兼容现有后端分页 API，无需改动
- 内存占用渐进增长，避免一次性加载大量数据

**备选方案**：一次性加载全部 → 首屏慢，大数据量时内存问题

### 2. 表头固定方案：CSS `position: sticky`

**选择**：使用 `thead { position: sticky; top: 0; }` 实现表头固定

**理由**：
- 原生 CSS 方案，性能好，无需 JavaScript 干预
- 现代浏览器（Electron）支持良好
- 保持表格语义完整

**备选方案**：
- 分离表头和内容为两个表格 → 列宽同步复杂
- 使用 `div` 模拟表格 → 失去表格语义和可访问性

### 3. 自定义进度条：独立组件 `ScrollProgressBar.vue`

**选择**：创建独立 Vue 组件，接收滚动容器 ref 和总/已加载数量

**组件接口**：
```vue
<ScrollProgressBar
  :scrollContainer="tableBodyRef"
  :totalItems="totalAlbums"
  :loadedItems="albums.length"
  @seek="handleSeek"
/>
```

**理由**：
- 独立组件便于测试和复用
- 通过 props 控制显示逻辑，与业务解耦

### 4. 进度条视觉与交互

**视觉设计**：
- 宽度：约 6px，圆角
- 颜色：滚动时 `rgba(150, 150, 150, 0.8)`，静止时 `rgba(150, 150, 150, 0.3)`
- 过渡：透明度变化 0.3s ease

**交互**：
- 鼠标拖动滑块：更新滚动位置
- 点击轨道任意位置：跳转到对应位置
- 隐藏原生滚动条：`::-webkit-scrollbar { display: none; }`

### 5. 无限滚动触发机制

**选择**：使用 `IntersectionObserver` 监听"哨兵元素"

**实现**：
- 在列表底部放置一个不可见的哨兵 `<div ref="sentinel">`
- 当哨兵进入视口时，加载下一页
- 加载时显示 spinner，加载完成或无更多数据时隐藏

**理由**：
- 比 `scroll` 事件监听更高效，不会频繁触发
- 现代浏览器原生支持，无需第三方库

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **[长列表性能]** 专辑数量 >1000 时 DOM 节点多 | 滚动卡顿 | 当前用户量级预计 <500，可接受；未来可升级虚拟滚动 |
| **[筛选/排序重置]** 用户更改筛选条件后需清空已加载数据 | 体验中断 | 重置时显示加载状态，快速加载首页 |
| **[滚动位置恢复]** 切换页面后返回，滚动位置丢失 | 需重新滚动 | 本次不实现位置记忆，接受此限制 |
